# РУМСЭТ — приветственный бот для MAX

Официальный чат-бот компании **РУМСЭТ** для мессенджера [MAX](https://max.ru). При запуске бот отправляет приветствие с кнопкой перехода в каталог продукции и контактную информацию для связи с менеджером.

---

## Что делает бот

1. Пользователь находит бота в MAX (через поиск или по ссылке) и нажимает «Начать».
2. Бот отправляет **приветственное сообщение** с описанием компании и кнопкой «Перейти в каталог продукции».
3. Через 1–2 секунды бот отправляет **второе сообщение** с телефонами и адресом шоурума.
4. На любое другое сообщение бот отвечает подсказкой использовать `/start`.

---

## Быстрый старт (локальный запуск)

### Что понадобится

- **Node.js** версии 18 или выше — [скачать здесь](https://nodejs.org/)
- **Git** — [скачать здесь](https://git-scm.com/downloads)
- **Токен бота MAX** — получается на [платформе партнёров MAX](https://partners.max.ru)

### Шаг 1. Клонирование проекта

> **Где выполнять:** в терминале (командная строка).
> **Что делает:** скачивает проект на ваш компьютер.

```bash
git clone <url-вашего-репозитория> roomset-max-bot
cd roomset-max-bot
```

**Ожидаемый результат:** появилась папка `roomset-max-bot` с файлами проекта.

### Шаг 2. Установка зависимостей

> **Где выполнять:** в терминале, внутри папки проекта.
> **Что делает:** скачивает все библиотеки, которые нужны боту для работы.
> **Зачем:** без этого шага код не запустится — ему нужны внешние модули.

```bash
npm install
```

**Ожидаемый результат:** появилась папка `node_modules`, в терминале нет ошибок.

**Если ошибка `npm not found`:** значит Node.js не установлен. Установите его с [nodejs.org](https://nodejs.org/) и перезапустите терминал.

### Шаг 3. Настройка переменных окружения

> **Где выполнять:** в файловом менеджере или терминале.
> **Что делает:** создаёт файл с настройками бота.
> **Зачем:** бот берёт токен, URL и другие параметры из этого файла. Это безопасный способ хранить секреты — файл `.env` не попадёт в Git.

```bash
cp .env.example .env
```

Затем откройте файл `.env` в любом текстовом редакторе и заполните:

```env
BOT_TOKEN=ваш_токен_бота_из_MAX
PORT=3000
PUBLIC_BASE_URL=https://ваш-публичный-url.com
```

> **Что такое env (переменные окружения)?**
> Это настройки, которые приложение читает при запуске. Они хранятся вне кода, чтобы секреты (токен бота) не попали в открытый доступ. Файл `.env` — это локальный файл с этими настройками.

**Обязательные переменные:**

| Переменная | Описание | Пример |
|---|---|---|
| `BOT_TOKEN` | Токен бота MAX | `AAHdqTcvCH1vGWJxfSeo...` |
| `PUBLIC_BASE_URL` | Публичный HTTPS URL | `https://your-app.up.railway.app` |

**Необязательные (есть значения по умолчанию):**

| Переменная | По умолчанию | Описание |
|---|---|---|
| `PORT` | `3000` | Порт HTTP-сервера |
| `WEBHOOK_PATH` | `/webhook/max` | Путь webhook-эндпоинта |
| `CATALOG_URL` | `https://room-set.ru/virtualnyj-shourum/` | URL каталога |
| `SECOND_MESSAGE_DELAY_MS` | `1500` | Задержка перед вторым сообщением (1000–2000 мс) |
| `LOG_LEVEL` | `info` | Уровень логов: `debug`, `info`, `warn`, `error` |
| `NODE_ENV` | `production` | Окружение: `development` или `production` |

### Шаг 4. Запуск в режиме разработки

> **Где выполнять:** в терминале, внутри папки проекта.
> **Что делает:** запускает бота с автоматической перезагрузкой при изменении кода.
> **Зачем:** удобно для разработки — не нужно перезапускать вручную.

```bash
npm run dev
```

**Ожидаемый результат:** в терминале появится:
```
Сервер запущен на порту 3000
Health-check доступен
```

### Шаг 5. Проверка работоспособности

> **Где выполнять:** в браузере или другом терминале.
> **Что делает:** проверяет, что сервер запущен и отвечает.

Откройте в браузере: `http://localhost:3000/health`

**Ожидаемый результат:**
```json
{ "status": "ok" }
```

## Настройка webhook

> **Что такое webhook?**
> Это способ, которым MAX «звонит» вашему боту. Когда пользователь пишет боту в MAX, платформа отправляет HTTP-запрос на ваш сервер. Для этого серверу нужен публичный HTTPS-адрес (URL), доступный из интернета.

### Как это работает

1. Вы запускаете бота на сервере (или локально через туннель).
2. При старте бот автоматически регистрирует свой URL в MAX.
3. MAX начинает отправлять события (нажатие «Начать», сообщения) на этот URL.

### Для локальной разработки

Для тестирования на локальном компьютере нужен туннель — сервис, который даёт временный публичный URL для вашего localhost. Например, [ngrok](https://ngrok.com/):

```bash
# Установка ngrok (если ещё не установлен)
# Скачайте с https://ngrok.com/download

# Запуск туннеля на порт 3000
ngrok http 3000
```

ngrok покажет публичный URL вида `https://abc123.ngrok-free.app`. Укажите его в `.env`:

```env
PUBLIC_BASE_URL=https://abc123.ngrok-free.app
```

Перезапустите бота — он зарегистрирует webhook автоматически.

### Как проверить, что webhook работает

1. Откройте бота в MAX.
2. Нажмите «Начать» или отправьте `/start`.
3. В терминале с ботом должны появиться логи:
   ```
   Получен webhook update
   Обработка события bot_started
   Приветственное сообщение отправлено успешно
   ```
4. В чате MAX должно появиться приветственное сообщение с кнопкой.

## Запуск через Docker

> **Что такое Docker?**
> Docker — программа, которая упаковывает приложение со всеми зависимостями в «контейнер». Контейнер работает одинаково на любом компьютере. Это как виртуальная коробка для вашего приложения.

> **Зачем Docker нужен здесь?**
> Чтобы не устанавливать Node.js и зависимости вручную на сервере. Достаточно установить Docker, и бот запустится одной командой.

### Что понадобится

- **Docker** — [скачать Docker Desktop](https://www.docker.com/products/docker-desktop/)

### Запуск

> **Где выполнять:** в терминале, внутри папки проекта.

```bash
# Создайте файл .env (если ещё не создан)
cp .env.example .env
# Заполните .env вашими данными (см. Шаг 3 выше)

# Соберите и запустите контейнер
docker compose up --build
```

**Что делает команда:**
1. `docker compose up` — запускает сервис, описанный в `docker-compose.yml`.
2. `--build` — пересобирает Docker-образ (нужно после изменений в коде).

**Ожидаемый результат:** бот запустился, в терминале видны логи.

**Остановка:**
```bash
docker compose down
```

**Если ошибка `docker: command not found`:** Docker не установлен. Скачайте [Docker Desktop](https://www.docker.com/products/docker-desktop/).

**Если ошибка при сборке:** убедитесь, что Docker Desktop запущен (иконка в трее).

## Как смотреть логи

### Локально (без Docker)

Логи выводятся прямо в терминал, где запущен бот. В режиме `development` логи цветные и человекочитаемые. В `production` — JSON-формат.

### В Docker

```bash
# Логи запущенного контейнера
docker compose logs -f
```

Флаг `-f` означает «следить» — новые логи будут появляться в реальном времени. Нажмите `Ctrl+C` чтобы выйти.

### Что означают уровни логов

| Уровень | Что означает |
|---|---|
| `info` | Нормальные события (запуск, отправка сообщений) |
| `warn` | Предупреждения (некритичные проблемы) |
| `error` | Ошибки (что-то пошло не так) |
| `debug` | Подробная информация (для отладки) |

## Типичные ошибки и что делать

### Приложение не стартует

**Симптом:** при `npm run dev` сразу появляется ошибка и процесс завершается.

**Решение:**
- Прочитайте текст ошибки. Если написано `Обязательная переменная окружения "BOT_TOKEN" не задана` — значит вы не создали или не заполнили файл `.env`.
- Убедитесь, что файл `.env` лежит в корне проекта (рядом с `package.json`).
- Убедитесь, что в `.env` нет лишних пробелов вокруг `=`.

### Не хватает env-переменных

**Симптом:** ошибка `Обязательная переменная окружения "..." не задана`.

**Решение:**
1. Откройте файл `.env`.
2. Сверьтесь с `.env.example` — все обязательные переменные должны быть заполнены.
3. `BOT_TOKEN` и `PUBLIC_BASE_URL` — обязательны.

### Webhook не получает события

**Симптом:** вы нажимаете «Начать» в MAX, но в логах бота ничего не появляется.

**Решение:**
1. Проверьте, что `PUBLIC_BASE_URL` в `.env` — это реальный публичный HTTPS URL, доступный из интернета.
2. Если работаете локально — используйте ngrok (см. раздел «Настройка webhook»).
3. Откройте `https://ваш-url/health` в браузере — должен вернуть `{"status":"ok"}`.
4. Проверьте логи — при старте бот пишет, успешно ли зарегистрирован webhook.
5. Если в логах `Не удалось зарегистрировать webhook` — проверьте `BOT_TOKEN`.

### Docker-образ не собирается

**Симптом:** ошибка при `docker compose up --build`.

**Решение:**
1. Убедитесь, что Docker Desktop запущен.
2. Убедитесь, что в папке проекта есть `Dockerfile`.
3. Попробуйте `docker compose down` и затем `docker compose up --build` заново.
4. Если ошибка связана с сетью — проверьте интернет-подключение.

### Railway не видит env

**Симптом:** бот задеплоен на Railway, но не работает.

**Решение:**
1. В дашборде Railway откройте ваш сервис → вкладка **Variables**.
2. Добавьте все переменные из `.env.example` (кроме `PORT` — Railway задаёт его автоматически).
3. Нажмите **Deploy** или дождитесь автоматического редеплоя.
4. Подробности — в файле `DEPLOY.md`.

### Бот не отвечает на /start

**Симптом:** webhook получает события (видно в логах), но сообщение не приходит.

**Решение:**
1. Проверьте логи — есть ли ошибки при отправке сообщения.
2. Если `401` — токен бота неверный или просрочен. Получите новый на платформе партнёров MAX.
3. Если `429` — слишком много запросов. Подождите и попробуйте снова.
4. Если `400` — возможно, изменился формат API MAX. Проверьте [документацию](https://dev.max.ru/docs-api).
